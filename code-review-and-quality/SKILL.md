# 代码审查与质量

## 概述

使用质量门禁进行多维度代码审查。每个变更在合并前都要审查——无一例外。审查涵盖五个维度：正确性、可读性、架构、安全性和性能。

**批准标准：** 当变更明确改善整体代码健康状况时批准它，即使它不完美。完美的代码不存在——目标是持续改进。不要因为变更不是你编写的方式而阻止它。如果它改善了代码库并遵循项目约定，就批准它。

## 何时使用

- 合并任何 PR 或变更之前
- 完成功能实现后
- 当另一个 agent 或模型产生了你需要评估的代码时
- 重构现有代码时
- 任何 bug 修复后（同时审查修复和回归测试）

## 五维度审查

每个审查都评估这些维度的代码：

### 1. 正确性

代码是否做了它声称要做的事情？
- 是否匹配规范或任务要求？
- 边缘情况是否处理（null、空、边界值）？
- 错误路径是否处理（不仅仅是 happy path）？
- 是否通过所有测试？测试是否实际测试了正确的内容？
- 是否存在偏移错误、竞争条件或状态不一致？

### 2. 可读性与简洁性

另一个工程师（或 agent）能否在没有作者解释的情况下理解这段代码？
- 名称是否描述性且与项目约定一致？（没有 `temp`、`data`、`result` 没有上下文）
- 控制流是否直接（避免嵌套三元运算符、深层回调）？
- 代码是否逻辑组织（相关代码分组，清晰的模块边界）？
- 是否有应该简化的"聪明"技巧？
- **能否用更少的行完成？**（1000 行而 100 行足够就是失败）
- **抽象是否值得它们带来的复杂度？**（不要在第三个用例之前泛化）
- 注释是否有助于澄清非显而易见的意图？（但不要注释显而易见的代码。）
- 是否有死代码工件：no-op 变量（`_unused`）、向后兼容垫片或 `// removed` 注释？

### 3. 架构

变更是否符合系统设计？
- 是否遵循现有模式或引入新模式？如果是新的，是否合理？
- 是否保持清晰的模块边界？
- 是否有应该共享的代码重复？
- 依赖是否沿正确方向流动（无循环依赖）？
- 抽象级别是否合适（不过度工程化，不太耦合）？

### 4. 安全性

有关详细安全指南，请参阅 `security-and-hardening`。变更是否引入漏洞？
- 用户输入是否经过验证和清理？
- 机密是否远离代码、日志和版本控制？
- 需要时是否检查了认证/授权？
- SQL 查询是否参数化（无字符串连接）？
- 输出是否编码以防止 XSS？
- 依赖是否来自可信来源且没有已知漏洞？
- 来自外部源的数据（API、日志、用户内容、配置文件）是否被视为不受信任？
- 外部数据流是否在用于逻辑或渲染之前在系统边界验证？

### 5. 性能

有关详细分析和优化，请参阅 `performance-optimization`。变更是否引入性能问题？
- 是否有 N+1 查询模式？
- 是否有无界循环或不受约束的数据获取？
- 是否有应该异步的同步操作？
- UI 组件中是否有不必要的重新渲染？
- 列表端点是否缺少分页？
- 热路径中是否创建了大对象？

## 变更大小

小而专注的变更更容易审查、合并更快、部署更安全。目标大小：

```
~100 行更改   → 好。可以一次审查。
~300 行更改   → 如果是单个逻辑变更则可接受。
~1000 行更改  → 太大。拆分它。
```

**什么算作"一个变更"：** 解决一个问题的单个独立修改，包含相关测试，并在提交后保持系统功能。功能的一部分——而不是整个功能。

**当变更太大时的拆分策略：**

| 策略 | 方法 | 时机 |
|------|------|------|
| **堆栈** | 提交一个小变更，基于它开始下一个 | 顺序依赖 |
| **按文件组** | 为需要不同审查者的组分开变更 | 横切关注点 |
| **水平** | 先创建共享代码/存根，然后是消费者 | 分层架构 |
| **垂直** | 将功能分解为更小的全栈切片 | 功能工作 |

**何时可以接受大变更：** 完整文件删除和自动化重构，审查者只需要验证意图，而不是每一行。

**将重构与功能工作分开。** 重构现有代码并添加新行为的变更是两个变更——分别提交它们。小清理（重命名变量）可以酌情包含在功能提交中。

## 变更描述

每个变更都需要一个在版本控制历史中独立存在的描述。

**第一行：** 简短、祈使、独立。"删除 FizzBuzz RPC" 而不是 "正在删除 FizzBuzz RPC"。必须信息丰富，以便搜索历史的人无需阅读差异就能理解变更。

**正文：** 正在更改什么以及为什么。包括代码本身不可见的上下文、决策和推理。链接到 bug 编号、基准结果或设计文档（如果相关）。承认方法缺点（如果存在）。

**反模式：** "修复 bug"、"修复构建"、"添加补丁"、"将代码从 A 移动到 B"、"第 1 阶段"、"添加便利函数"。

## 审查过程

### 步骤 1：理解上下文

在查看代码之前，理解意图：
- 这个变更试图完成什么？
- 它实现了什么规范或任务？
- 预期的行为变更是什么？

### 步骤 2：先审查测试

测试揭示意图和覆盖范围：
- 变更是否有测试？
- 它们是否测试行为（而不是实现细节）？
- 边缘情况是否覆盖？
- 测试是否有描述性名称？
- 如果代码更改，测试是否会捕获回归？

### 步骤 3：审查实现

带着五个维度审视代码：

```
对于每个更改的文件：
1. 正确性：此代码是否做了测试所说它应该做的？
2. 可读性：我能在没有帮助的情况下理解这个吗？
3. 架构：这是否符合系统？
4. 安全性：有任何漏洞吗？
5. 性能：有任何瓶颈吗？
```

### 步骤 4：分类发现

为每个评论标记其严重性，以便作者知道什么是必需的 vs 可选的：

| 前缀 | 含义 | 作者操作 |
|------|------|----------|
| *（无前缀）* | 必需的更改 | 合并前必须解决 |
| **关键：** | 阻止合并 | 安全漏洞、数据丢失、功能损坏 |
| **Nit：** | 小的，可选的 | 作者可以忽略——格式化、风格偏好 |
| **可选：** / **考虑：** | 建议 | 值得考虑但不是必需的 |
| **FYI** | 仅供参考 | 无需操作——供将来参考的上下文 |

这防止作者将所有反馈视为强制性的，并在可选建议上浪费时间。

### 步骤 5：验证验证

检查作者的验证故事：
- 运行了哪些测试？
- 构建是否通过？
- 变更是否手动测试？
- UI 变更是否有截图？
- 是否有前后比较？

## 多模型审查模式

使用不同模型进行不同审查视角：

```
模型 A 编写代码
    │
    ▼
模型 B 审查正确性和架构
    │
    ▼
模型 A 处理反馈
    │
    ▼
人类做出最终决定
```

这捕获了单个模型可能遗漏的问题——不同模型有不同的盲点。

**审查 agent 的示例提示：**
```
审查此代码变更以确保正确性、安全性和对我们项目约定的遵循。
规范说 [X]。变更应该 [Y]。
将任何问题标记为关键、重要或建议。
```

## 死代码卫生

在任何重构或实现变更后，检查孤立代码：
1. 识别现在不可达或未使用的代码
2. 明确列出它
3. **删除前询问：** "我应该删除这些现在未使用的元素吗：[列表]？"

不要留下死代码——它会混淆未来的读者和 agent。但不要默默删除你不理解的东西。有疑问时，询问。

```
已识别死代码：
- src/utils/date.ts 中的 formatLegacyDate() — 被 formatDate() 替换
- src/components/ 中的 OldTaskCard 组件 — 被 TaskCard 替换
- src/config.ts 中的 LEGACY_API_URL 常量 — 没有剩余引用
→ 可以安全删除这些吗？
```

## 审查速度

慢审查会阻碍整个团队。上下文切换到审查的成本小于强加给其他人的等待成本。

- **在一个工作日内响应** — 这是最大值，不是目标
- **理想节奏：** 在审查请求到达后不久响应，除非深处于专注编码中。典型的变更应该在一天内完成多轮审查
- **优先快速个人响应** 而不是快速最终批准。快速反馈即使需要多轮也能减少挫败感
- **大变更：** 要求作者拆分它们，而不是审查一个巨大的变更集

## 处理分歧

解决审查争议时，应用此层次结构：

1. **技术事实和数据** 覆盖意见和偏好
2. **样式指南** 是风格问题的绝对权威
3. **软件设计** 必须基于工程原则而非个人偏好进行评估
4. **代码库一致性** 如果不降低整体健康状况则是可接受的

**不要接受"我稍后会清理它。"** 经验表明延迟清理很少发生。要求在提交前清理，除非是真正的紧急情况。如果周围问题不能在此变更中解决，要求提交一个带有自我分配的 bug。

## 审查中的诚实

审查代码时——无论是你自己、另一个 agent 还是人类编写的：
- **不要橡皮图章。** 没有审查证据的 "LGTM" 对任何人都没有帮助。
- **不要淡化真正的问题。** 当它是一个将击中生产的 bug 时说"这可能是个小问题"是不诚实的。
- **尽可能量化问题。** "这个 N+1 查询将为列表中的每个项目增加约 50ms" 比"这可能很慢"更好。
- **对有明显问题的方法进行反驳。** 谄媚是审查中的失败模式。如果实现有问题，直接说出来并提出替代方案。
- **优雅地接受覆盖。** 如果作者有完整上下文并不同意，遵从他们的判断。评论代码，而不是人——将个人批评重新构建为专注于代码本身。

## 依赖纪律

代码审查的一部分是依赖审查：

**添加任何依赖之前：**
1. 现有技术栈解决这个问题吗？（通常可以。）
2. 依赖有多大？（检查包影响。）
3. 它是否积极维护？（检查最后提交、开放问题。）
4. 它是否有已知漏洞？（`npm audit`）
5. 许可证是什么？（必须与项目兼容。）

**规则：** 优先使用标准库和现有实用程序而不是新依赖。每个依赖都是负担。

```markdown
## 审查：[PR/变更标题]

### 上下文
- [ ] 我理解这个变更做什么以及为什么

### 正确性
- [ ] 变更匹配规范/任务要求
- [ ] 边缘情况已处理
- [ ] 错误路径已处理
- [ ] 测试充分覆盖变更

### 可读性
- [ ] 名称清晰且一致
- [ ] 逻辑直接
- [ ] 无不必要的复杂度

### 架构
- [ ] 遵循现有模式
- [ ] 无不必要的耦合或依赖
- [ ] 适当的抽象级别

### 安全性
- [ ] 代码中没有机密
- [ ] 输入在边界处验证
- [ ] 无注入漏洞
- [ ] 认证检查到位
- [ ] 外部数据源被视为不受信任

### 性能
- [ ] 无 N+1 模式
- [ ] 无无界操作
- [ ] 列表端点有分页

### 验证
- [ ] 测试通过
- [ ] 构建成功
- [ ] 已完成手动验证（如适用）

### 判定
- [ ] **批准** — 准备合并
- [ ] **请求更改** — 问题必须解决
```

## 常见误解

| 误解 | 现实 |
|------|------|
| "它工作了，那就够了" | 不可读、不安全或架构错误的工作代码会产生复合债务。 |
| "我写的，所以我知道它是正确的" | 作者对自己的假设视而不见。每个变更都受益于另一双眼睛。 |
| "我们稍后会清理它" | 后来永远不会到来。审查是质量门禁——使用它。要求在合并前清理，而不是之后。 |
| "AI 生成的代码可能没问题" | AI 代码需要更多审查，而不是更少。它自信且合理，即使在错误时也是如此。 |
| "测试通过了，所以很好" | 测试是必要的但不充分。它们不捕获架构问题、安全性问题或可读性问题。 |

## 红旗

- 没有任何审查就合并的 PR
- 只检查测试是否通过的审查（忽略其他维度）
- 没有实际审查证据的 "LGTM"
- 没有安全重点审查的安全敏感变更
- "太大无法正确审查" 的大 PR（拆分它们）
- 没有回归测试的 bug 修复 PR
- 没有严重性标签的审查评论——不清楚什么是必需的 vs 可选的
- 接受"我稍后会修复它"——它永远不会发生

## 验证

审查完成后：

- [ ] 所有关键问题已解决
- [ ] 所有重要问题已解决或明确推迟并说明理由
- [ ] 测试通过
- [ ] 构建成功
- [ ] 验证故事已记录（更改了什么，如何验证）
